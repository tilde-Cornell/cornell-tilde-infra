# tilde@Cornell System Overview

> Generated by Codex using GPT-5.5 with extra-high intelligence and manually reviewed by Korra Littwin.

This repository is infrastructure for a tilde-style Cornell community server. The server publishes static pages with Apache, gives approved members Unix accounts, lets each member edit a personal `~/public_html` site, and maintains a generated public member directory.

The intended production checkout model is unusual but deliberate: the Git worktree is checked out at `/`, so repository paths map directly onto their runtime locations. For example, `deploy/setup.sh` becomes `/deploy/setup.sh`, `opt/cornell-tilde/bin/approve_user.py` becomes `/opt/cornell-tilde/bin/approve_user.py`, and `var/www/html/index.html` becomes `/var/www/html/index.html`.

## Main Runtime Paths

- `/deploy`: installer and deployment scripts.
- `/opt/cornell-tilde/bin`: operational tools and application/account scripts.
- `/opt/cornell-tilde/lib/cornell_tilde`: shared Python configuration and database helpers.
- `/opt/cornell-tilde/migrations`: SQLite schema setup files and migrations.
- `/opt/cornell-tilde/systemd`: systemd units for rebuilding the public directory.
- `/opt/cornell-tilde/templates`: HTML templates for generated member pages.
- `/opt/cornell-tilde/var/cornell_tilde.sqlite3`: live SQLite database, not stored in Git.
- `/var/www/html`: Apache document root and public static site.
- `/home/<username>/public_html`: each user’s personal website directory.

## High-Level Components

Apache serves the public site from `/var/www/html` and member pages through `mod_userdir`. SSH is used both for ordinary member access and for the public `join` account. The `join` account is locked into the terminal application form by an SSH `ForceCommand`.

The application flow stores pending applications in SQLite. An admin runs `approve_user.py` to review applications, create Unix accounts, write SSH authorized keys, generate starter webpages, and add approved users to the public directory table.

Directory pages are static HTML generated from the `users` table. SQLite triggers set a `directory_modified` flag when relevant user rows change. A systemd path unit watches the database file and starts a oneshot service that regenerates the directory when the flag is set.

## Security Boundary

The public `join` account should not have direct database group access. The intended boundary is:

- `join` can read and execute only the join form code it needs.
- `join` can run `/opt/cornell-tilde/bin/submit_application.py` through sudo without a password.
- `submit_application.py` validates input and writes only a pending application.
- Admin-only tools and direct database access remain root/admin operational actions.

## Operational Model

Initial setup is handled by `/deploy/setup.sh`. Routine updates should be handled by pulling the selected branch into the root worktree and then running `sudo post-deploy`. The post-deploy path reapplies runtime permissions, recreates symlinks, reruns database initialization/migrations, reloads systemd state, and verifies basic service health.

This split matters because Git only updates files. It does not automatically restore file ownership, ACLs, service links, database schema changes, or systemd state.
