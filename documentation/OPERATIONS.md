# Operations Guide

> Generated by Codex using GPT-5.5 with extra-high intelligence and manually reviewed by Korra Littwin.

This guide covers the common operational tasks after the server has been installed.

## Deploying Updates

After setup, use the root Git worktree alias and rerun post-deploy:

```bash
prodgit pull origin development
sudo post-deploy
```

For production main:

```bash
prodgit pull origin main
sudo post-deploy
```

`post-deploy` calls `apply-runtime.sh`, which reapplies the runtime state that Git cannot handle by itself:

- Runtime directories.
- Database initialization and compatibility schema updates.
- Ownership, modes, and ACLs.
- Restricted `join` account access.
- Sudoers rule for application submission.
- Admin command symlinks in `/usr/local/sbin`.
- systemd service/path symlinks and daemon reload.
- Directory rebuild trigger.
- Basic Apache, SSH, sudoers, database, and systemd verification.

## Fresh Setup

Use `documentation/SETUP.md` for the full first-server checklist. At a high level:

1. Create an initial sudo admin.
2. Add a root deploy key for GitHub.
3. Check out this repo as a root worktree at `/`.
4. Run `sudo /deploy/setup.sh`.
5. Run Certbot manually.
6. Test `ssh join@<domain>`.

## Main Commands

Run the health/admin check:

```bash
sudo tilde-admin
```

Review applications:

```bash
sudo approve_user.py
```

Force a directory rebuild:

```bash
sudo systemctl reset-failed cornell-tilde-directory.service
sudo systemctl start cornell-tilde-directory.service
```

Check the directory watcher:

```bash
systemctl status cornell-tilde-directory.path --no-pager
```

Check recent directory rebuild logs:

```bash
journalctl -u cornell-tilde-directory.service --no-pager -n 80
```

Check SSH config validity before restarting SSH:

```bash
sudo sshd -t
```

Check Apache config validity:

```bash
sudo apache2ctl configtest
```

## Important Files to Inspect During Incidents

- `/etc/ssh/sshd_config`: SSH hardening and the managed `join` forced-command block.
- `/etc/sudoers.d/join_script`: restricts `join` to the submit helper.
- `/opt/cornell-tilde/.env`: server domain and admin contact email.
- `/opt/cornell-tilde/var/cornell_tilde.sqlite3`: live database.
- `/var/www/html/directory/index.html`: generated public directory landing page.
- `/home/<username>/.ssh/authorized_keys`: approved user SSH key.
- `/home/<username>/public_html/index.html`: approved user webpage.

## Permissions Model

The code and templates under `/opt/cornell-tilde` are root-owned. The database and var directory are owned by `root:cornelltilde-db` with restricted group access. The `join` account should not be a member of `cornelltilde-db`; it submits applications through the root-owned helper instead.

Approved user home directories are set to allow Apache traversal and public page serving while keeping SSH key material restricted:

- home directory: `711`
- `.ssh`: `700`
- `authorized_keys`: `600`
- `public_html`: `755`
- starter `index.html`: `644`

## Troubleshooting Notes

If `sudo post-deploy` is missing, run the deploy script by absolute path once to recreate command symlinks:

```bash
sudo bash /deploy/apply-runtime.sh
```

If the directory service reports `start-limit-hit`, reset it and start it once:

```bash
sudo systemctl reset-failed cornell-tilde-directory.service
sudo systemctl start cornell-tilde-directory.service
```

If `prodgit pull` refuses because a local file changed, inspect the file before overwriting anything:

```bash
prodgit status
prodgit diff -- <path>
```
