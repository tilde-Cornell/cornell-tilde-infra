# Application and Account Flow

> Generated by Codex using GPT-5.5 with extra-high intelligence and manually reviewed by Korra Littwin.

This document explains how a prospective member gets from the public `join` SSH account to an approved Unix account with a personal webpage.

## Public Join Entry

Applicants connect over SSH as the shared `join` user:

```bash
ssh join@<server-domain>
```

The SSH server should force that account to run:

```bash
/opt/cornell-tilde/bin/join_script_wrapper.sh
```

The wrapper starts `join_script.py`, which displays the terminal application form. The applicant provides:

- Cornell email address.
- Full name.
- Requested username.
- Cornell college or school.
- Graduation year.
- Additional information for the reviewer.
- SSH public key.

The join UI performs friendly validation, but the trusted validation happens later in `submit_application.py`.

## Application Submission

`join_script.py` serializes the application as JSON and invokes:

```bash
sudo /opt/cornell-tilde/bin/submit_application.py
```

The sudoers rule should allow the `join` user to run only that helper. The helper enforces payload size limits, field length limits, Cornell email format, username format, accepted college values, four-digit graduation year format, and one-line SSH public key format.

Application IDs use this format:

```text
CT-YY-XXXXXX
```

The suffix avoids ambiguous characters and is retried up to 20 times if SQLite reports an ID collision.

## Admin Review

Admins review pending applications with:

```bash
sudo approve_user.py
```

The approval tool shows pending applications, warns about duplicates against active users, and lets the admin approve, reject, move next/previous, refresh, or quit.

When approving, the admin can accept or change the requested username. Username validation currently allows:

- 2 to 32 characters.
- Lowercase letters, numbers, hyphen, and underscore.
- First character must be a lowercase letter.

## Account Creation

On approval, `approve_user.py`:

1. Creates a Unix user with `adduser --disabled-password`.
2. Writes the applicant’s SSH public key to `/home/<username>/.ssh/authorized_keys`.
3. Creates `/home/<username>/public_html`.
4. Generates the starter `index.html` from `/opt/cornell-tilde/templates/user_homepage.html`.
5. Sets ownership and permissions on the home, `.ssh`, authorized keys, and public site files.
6. Adds the user to the SQLite `users` table.
7. Marks the application as approved with the final username.
8. Prints a welcome message for the admin to send manually.

If account creation partially succeeds and a later step fails, the tool attempts rollback by removing the user from the directory table and deleting the Unix account/home directory.

## Rejection

Rejecting an application marks it as `rejected` in the database and prints a rejection message for manual sending. Rejected applications remain in the database for record keeping but no longer appear in pending review.

## Member Websites

Approved users publish webpages by editing:

```bash
~/public_html/index.html
```

Apache serves those pages through `mod_userdir` at:

```text
https://<server-domain>/~<username>
```

This setup is designed for static HTML/CSS/JS pages. User-run long-lived web services, WebSockets, or Socket.IO apps are not automatically supported by the current Apache/userdir configuration.
